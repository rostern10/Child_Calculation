# AGENTS.md

本文件用于约束后续在本仓库中的开发、修改、验证与部署方式。  
适用范围：当前 `nutrition-eval-web` 整个项目。

## 1. 项目定位

- 这是一个院内 / 内网使用的纯前端静态评估网页。
- 默认部署环境是 Windows + IIS。
- 项目必须支持离线运行。
- 不依赖后端服务、不依赖数据库、不依赖联网接口。

## 2. 页面与交互原则

- 页面风格保持白底、宋体、大字号、信息清晰。
- 主页面和附录页面分开，菜单保持极简。
- 结果展示优先“一眼看懂”，不要在主结果区堆过多说明文字。
- 评估结果尽量在当前卡片内展开，不要改成全屏铺满式输出。
- 日期输入优先稳定、兼容性好的原生方案，不优先使用复杂日历组件。

## 3. 浏览器兼容要求

- 默认要考虑老版本 Chrome / Win7 场景。
- 尽量避免使用兼容性差的新语法或新 API。
- 修改前端代码时，优先选择原生、简单、低依赖实现。
- 任何新交互都要优先保证“能显示、能输入、能提交、能看清”。

## 4. 业务分段规则

### 7 岁以下

- `7 岁以下` 与 `7 岁及以上` 使用两套不同的评估逻辑。
- `7 岁以下` 输出两张结果表：
  - `表 1：儿童生长水平评价`
  - `表 2：儿童营养状况评价`
- 涉及指标：
  - 年龄别体重
  - 年龄别身长 / 身高
  - 身长 / 身高别体重
  - 年龄别 BMI

### 7 岁及以上

- 年龄切换点按 `84 月龄` 处理。
- `7 岁及以上` 不输出 7 岁以下两张表，改为输出 3 项结果：
  - 年龄身高筛查生长迟缓等级结果
  - 年龄 BMI 筛查营养状态结果
  - 身高发育等级判定结果

## 5. 边界月龄规则

- `7 岁以下` 实际支持到 `83 月龄`，即 `6 岁 11 月`。
- `6 岁 10 月`、`6 岁 11 月` 的年龄别指标按 7 岁以下最后一档沿用。
- 遇到年龄边界时，必须先确认标准表实际覆盖范围，再决定是否沿用最后一档。

## 6. 数据来源规则

- 所有评估结论必须以本地标准表数据为准，不能凭经验写死。
- 新增业务规则时，顺序必须是：
  1. 先补标准数据文件
  2. 再改评估逻辑
  3. 再补附录展示
  4. 最后补测试
- 7 岁以下与 7 岁及以上的数据源分开维护。

## 7. 代码结构约定

- 主页面输入与展示逻辑：`app.mjs`
- 评估核心逻辑：`lib/evaluator.mjs`
- 标准数据：`data/*.mjs`
- 附录导航页：`appendix-index.html`
- 附录查看页：`appendix-viewer.html` + `appendix-viewer.mjs`
- 附录原始资料：`appendix/`

新增评估指标时，优先沿用现有结构，不要把规则散写在页面模板中。

## 8. 附录维护规则

- 附录页用于展示“指标依据”，主页面用于展示“最终结果”。
- 不要把附录说明大段搬进主结果区。
- 男童 / 女童、不同年龄段的表格可以按“同一指标合并展示”的方式整理。
- 附录命名应尽量直观，便于业务人员查找。

## 9. 部署规则

- 生产更新优先整包替换 `nutrition-eval-web` 目录。
- `web.config` 必须保留。
- 部署后要考虑浏览器缓存，必要时提醒使用 `Ctrl + F5`。
- 如果只更新部分文件，至少同步：
  - `index.html`
  - `app.mjs`
  - `lib/evaluator.mjs`
  - `data/`
  - `appendix-index.html`
  - `appendix-viewer.html`
  - `appendix-viewer.mjs`
  - `appendix/`

## 10. 修改后的验证要求

每次修改评估逻辑或标准表后，至少完成以下验证：

1. 跑样例测试。
2. 做页面真实录入验证。
3. 至少覆盖：
   - 1 组 `7 岁以下`
   - 1 组 `7 岁及以上`
   - 1 组边界值或异常值
4. 验证时不仅看页面结果，还要对照阈值表确认“为什么是这个结果”。

## 11. README 与 AGENTS 的分工

- `README.md`：对外说明项目用途、运行方式、部署方式。
- `AGENTS.md`：对内约束项目开发规则、业务边界、兼容性要求、验证要求。

后续如果两者内容冲突，以更适合其职责的那份为准：

- 对外介绍看 `README.md`
- 开发约束看 `AGENTS.md`
